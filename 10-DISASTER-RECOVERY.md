# DISASTER-RECOVERY — แผนกู้คืนระบบฉุกเฉิน
## BRS: XAMPP Backup & Restore System

---

## 1. RTO / RPO เป้าหมาย

| ระดับความสำคัญระบบ | RPO (ข้อมูลสูญหายได้ไม่เกิน) | RTO (เวลากู้คืนไม่เกิน) | ความถี่ backup แนะนำ |
|---|---|---|---|
| Critical (เช่น HR2000, ERP หลัก) | 4 ชั่วโมง | 30 นาที | ทุก 4-6 ชั่วโมง |
| Important (ระบบ BU ทั่วไป) | 24 ชั่วโมง | 1 ชั่วโมง | ทุกวัน (daily) |
| Low Priority (ระบบทดสอบ/internal tool) | 7 วัน | 4 ชั่วโมง | ทุกสัปดาห์ |

> ตั้งค่า `schedule_cron` ของแต่ละ Job ให้สอดคล้องกับระดับความสำคัญของระบบนั้นๆ

## 2. สถานการณ์ฉุกเฉินที่ต้องเตรียมรับมือ

### 2.1 เว็บแอปเสียหาย/ถูกลบโดยไม่ตั้งใจ (ไฟล์เท่านั้น ฐานข้อมูลปกติ)

```
1. เข้า Web UI → เลือก Job → History → เลือก backup version ล่าสุดที่ verification_status = passed
2. กด Restore → เลือกโหมด "Real Restore" → restore_target = "original"
3. ระบุ confirm_job_name ยืนยัน
4. ระบบจะ:
   - สร้าง Pre-Restore Snapshot อัตโนมัติ (เผื่อ rollback)
   - decrypt + ตรวจ checksum
   - extract ไฟล์ทับ path เดิม
5. ตรวจสอบเว็บแอปทำงานปกติหลัง restore
```
**เวลาประมาณการ:** 5-15 นาที สำหรับระบบขนาด < 2GB

### 2.2 ฐานข้อมูลเสียหาย/ข้อมูลถูกลบผิดพลาด

```
1. หยุดการเขียนข้อมูลเพิ่มเติมที่ฐานข้อมูลปลายทางทันที (ปิด service ที่เกี่ยวข้องชั่วคราว ถ้าจำเป็น)
2. เลือก backup version ที่ต้องการ (เลือกช่วงเวลาก่อนเกิดปัญหา ไม่จำเป็นต้องเป็นล่าสุดเสมอไป)
3. Restore โดยเลือก backup_type ที่มีเฉพาะฐานข้อมูล หรือ restore แบบ both ก็ได้ (ระบบ restore เฉพาะส่วนที่มีในไฟล์)
4. ตรวจสอบจำนวน table/record หลัง restore เทียบกับ manifest.json
```

### 2.3 เซิร์ฟเวอร์ล่มทั้งเครื่อง (ต้องย้ายไปเครื่องใหม่)

```
1. ติดตั้ง XAMPP + BRS บนเครื่องใหม่ตาม DEPLOYMENT-GUIDE.md
2. คัดลอก encryption.key สำรองที่เก็บไว้นอกระบบมาวางใน config/
3. เชื่อมต่อ storage target เดิม (เช่น NAS/Cloud) เข้ากับ BRS เครื่องใหม่
4. Import การตั้งค่า Job จาก JSON export ล่าสุด (ถ้ามี) หรือสร้างใหม่
5. เลือก backup version ล่าสุดจาก storage target → Restore แบบ Real ไปยัง path ใหม่
6. ปรับ DNS/host ให้ชี้มาที่เครื่องใหม่
```
**เวลาประมาณการ:** 1-3 ชั่วโมง ขึ้นกับขนาดข้อมูลและความเร็วเครือข่าย

### 2.4 Backup เสียหาย (Verification ล้มเหลว)

```
1. ระบบแจ้งเตือนทันทีผ่าน LINE Notify เมื่อพบ (อย่ารอให้ค้นพบตอนต้องการ restore จริง)
2. ตรวจสอบ backup version ก่อนหน้าที่ verification_status = passed แทน
3. ตรวจสอบสาเหตุ: disk เสีย / network ขาดระหว่างอัปโหลด / encryption key ผิด
4. รัน backup ใหม่ทันทีหลังแก้สาเหตุ เพื่อลด RPO gap
```

### 2.5 Restore แล้วพบว่าเลือกผิดเวอร์ชัน/ผิดพลาด

```
1. ใช้ Pre-Restore Snapshot ที่ระบบสร้างอัตโนมัติก่อน restore ทุกครั้ง
2. กด "Rollback" ใน Web UI (หรือ restore จาก snapshot นั้นซ้ำอีกครั้งผ่านขั้นตอนปกติ)
3. ระบบกลับสู่สถานะก่อนการ restore ที่ผิดพลาด
```

## 3. Restore Drill — การซ้อมกู้คืนระบบ (สำคัญที่สุด)

> **กฎเหล็ก: "Backup ที่ไม่เคยทดสอบ restore ไม่ใช่ backup ที่เชื่อถือได้"**

### ตารางการซ้อม (แนะนำ)

| ความถี่ | กิจกรรม | ผู้รับผิดชอบ |
|---------|---------|--------------|
| ทุกครั้งที่สร้าง Job ใหม่ | ทดสอบ Dry-Run Restore ทันทีหลังสร้าง backup แรก | ผู้สร้าง Job |
| รายเดือน | สุ่มเลือก 1 Job ทำ Real Restore ไปยัง path/database ทดลอง (alternate target) ตรวจสอบความถูกต้อง | IT Admin |
| รายไตรมาส | ซ้อมสถานการณ์ "เซิร์ฟเวอร์ล่มทั้งเครื่อง" บนเครื่องทดสอบจริง วัดเวลาที่ใช้เทียบกับ RTO เป้าหมาย | IT Admin + System Owner |

### Checklist การซ้อมแต่ละครั้ง
- [ ] เลือก backup version ที่จะทดสอบ (ไม่จำเป็นต้องเป็นล่าสุด)
- [ ] Restore ไปยัง alternate path/database (ไม่ทับของจริง)
- [ ] ตรวจสอบไฟล์ครบตามจำนวนใน manifest.json
- [ ] ตรวจสอบฐานข้อมูล: จำนวน table ตรง, สุ่มตรวจ record สำคัญ
- [ ] ทดสอบเปิดเว็บแอปที่ restore แล้วใช้งานได้จริง (login ได้, หน้าหลักโหลดได้)
- [ ] บันทึกเวลาที่ใช้ทั้งหมดเทียบกับ RTO เป้าหมาย
- [ ] ลบข้อมูลทดสอบ (alternate path/database) หลังซ้อมเสร็จ

## 4. Emergency Contact & Escalation

| ลำดับ | บทบาท | หน้าที่เมื่อเกิดเหตุฉุกเฉิน |
|-------|--------|------------------------------|
| 1 | IT Admin (เจ้าของระบบ BRS) | ดำเนินการ restore, ตัดสินใจเลือก backup version |
| 2 | System Owner ของ BU ที่ได้รับผลกระทบ | ยืนยันความถูกต้องของข้อมูลหลัง restore |
| 3 | ผู้บริหาร/ผู้มีอำนาจตัดสินใจ | กรณีต้อง downtime นานเกินที่ยอมรับได้ หรือข้อมูลสูญหายเกิน RPO |

แนะนำให้สร้าง LINE Group แยกสำหรับ "BRS Critical Alert" ที่ทุก storage target failure, verification failure, และ restore event จะถูกส่งเข้ากลุ่มนี้ทันที เพื่อให้ทีมรับรู้พร้อมกัน
